## 0. 定位原则（先读，避免走偏）

这些原则是 2026-06-11 方向讨论的结论，用于约束后续投入方向，**优先级高于任何具体功能**：

1. **多模态 RAG 的价值在「摄取/理解端不丢信息」，不在「检索端跨模态花活」。**
2. **真需求**：表格/公式/版式/图表信息能进入答案；答案命中后附**证据图**。
3. **伪需求 / 被高估**：把「以文搜图 / 图搜文」当独立功能；`query` 传图做跨模态向量检索（路径B）在通用 RAG 里需求窄。
4. RAG 形态演进：Naive → **Advanced/Modular（本项目所在，基本盘）** → **Agentic RAG（当前最热加分项）**。
5. 取舍：把力气投到 **P0 摄取质量** 和 **P1 Agentic RAG**，**不要**再为以文搜图等伪需求加重投入。

## 1. 优先级总览

| 级别 | 项 | 一句话价值 | 预估工作量 |
|---|---|---|---|
| **P0** | PDF 版面感知解析 | 修断词+阅读顺序，连纯文本检索一起受益 | 中 |
| **P0** | 表格结构化抽取 | 财报/论文表格数据不再丢 | 中 |

| P2 | 公式提取（LaTeX） | 学术场景刚需，通用可缓 | 中 |

## 2. P0 — 摄取端文档解析升级（最高性价比）

### 2.1 版面感知解析（修断词 + 阅读顺序）★最该先做

- **问题**：当前仅用 markitdown，双栏学术 PDF 出现：
  - 单词粘连丢空格：实测 `around96%ofourdataset`、`weestimatethattheactualfraction`
  - 双栏阅读顺序错乱、表格碎成 `| | | |`
  - 后果：不仅丢信息，**粘连词还拖累纯文本 embedding 质量**。
- **方案**：引入版面感知解析，替换或前置于 markitdown：
  - 轻量：`PyMuPDF (fitz)` 按 block/坐标重排，修复双栏顺序与空格；
  - 进阶：`unstructured`（hi_res 版面分区）或专攻学术 PDF 的 `MinerU` / `marker`。
- **落点**：`src/libs/loader/pdf_loader.py`（保持 BaseLoader 接口；可作为新 provider，配置可切）。
- **验收**：
  - 抽样 chunk 无明显粘连词（空格保留率显著提升）；
  - 同一 query 的检索相关性较现状提升（人工对比 top5）；
  - 现有测试 328 不回归。
- **风险**：新解析依赖体积/速度；用配置开关，保留 markitdown 回退。

### 2.2 表格结构化抽取

- **问题**：实测搜「BLEU 结果表」召回的是普通段落，**表格数字矩阵丢失**。
- **方案**：
  - `pdfplumber` / `camelot` 单独检测并抽取表格 → 转 **Markdown/HTML 表格**（保留行列语义）作为独立 chunk；
  - 或对表格区域用 VLM（qwen-vl-max）转结构化文本（成本高，质量好）。
- **落点**：loader 抽表 + 新 transform 或在 chunker 中以「表格块」单独成 chunk，`metadata.block_type="table"`。
- **验收**：财报/论文表格的关键数字能被检索命中并出现在答案中。
- **风险**：表格检测误判；跨页表格合并。

### 2.3 图 caption 定位 + 图注关联（提质，已做了图，补这步）

- **问题**：现在所有 `[IMAGE: id]` 占位**集中追加在文末** → caption 全堆到最后一个 chunk，与原文位置脱节；未利用 PDF 里的 `Figure N: ...` 图注。
- **方案**：
  - 用 `PyMuPDF` 拿到图片在页面的坐标，把占位插到**对应位置**而非文末；
  - 提取图下方/附近的 **图注文本（Figure N: ...）**，与图绑定，拼进图记录的 `text`（图注是最精准的图说明）。
- **落点**：`pdf_loader.py`（占位定位 + 图注提取）、`image_encoder.py`（图注入 text）。
- **验收**：含图 chunk 的 `image_refs` 与正文位置匹配；图记录 text 含图注。

### 2.4 结构感知 chunking（配套）

- **问题**：纯字符 `recursive` 切分会切碎表格/标题/段落。
- **方案**：按标题层级、段落、表格/图块边界切分（已有 `document_structure` splitter 可启用/增强）。
- **落点**：`src/ingestion/chunking/` + splitter 配置。

## 3. P1 — Agentic RAG（最热加分项，单独里程碑）

把「一次性检索→生成」升级为「**LLM 作为 agent 主动决策检索**」。建议分步落地：

| 步骤 | 能力 | 说明 |
|---|---|---|
| 3.1 | **检索决策（route）** | LLM 先判断「要不要检索 / 查哪个 collection」，简单问题直答 |
| 3.2 | **查询改写 / 分解** | 把复杂问题拆成多个子查询，分别检索再汇总 |
| 3.3 | **多跳检索（multi-hop）** | 基于首轮结果决定是否追加检索（迭代式） |
| 3.4 | **self-correction / 反思** | 评估检索结果是否足够，不足则重检（参考 Self-RAG / CRAG 思路） |
| 3.5 | **工具化** | 把检索封装成 agent 可调用的 tool（与现有 MCP 工具天然契合） |

- **落点**：新增 `src/core/agent/`（agent 编排），复用现有 `HybridSearch` / `LLMFactory` / `TraceCollector`。
- **验收**：在多跳问题上对比传统 RAG，答案完整度/正确率提升；trace 能看到多步检索决策。
- **风险**：延迟与 token 成本上升；需做循环上限与降级。
- **面试价值**：★★★★★（能讲清 Agentic RAG 设计是当前最稀缺的亮点）。

## 4. P2 / P3 — 其他

- **P2 公式提取**：`nougat` / `pix2tex` 提 LaTeX，或 VLM 转述；学术场景刚需。
- **P3 ColPali / ColQwen 视觉 late-interaction**：整页当图编码、绕过解析，是多模态文档 RAG 前沿 SOTA。作为「下一代摄取」预研，可写进项目 roadmap 作为前沿延伸（面试加分），不必现在落地。

---

## 5. 明确「不做 / 降级」清单（伪需求，避免浪费）

- ❌ 不为「**独立的以文搜图 / 图搜文**」加重投入（demo 价值 > 实用价值）。
- ❌ 不把「`query` 传图做跨模态向量检索（路径B）」当核心；如需 query 传图，真实形态是 **路径A（VLM 看懂图 → 文本检索）**。
- ⬇️ 现有「图片专项检索通道 / `modality=image` 过滤」可作为**可选能力**保留展示，不投入优化。
- ✅ 保留 **resp 返证据图**（低成本、提升可信度），定位为「证据」而非卖点。

## 6. 建议推进顺序与里程碑（下次从这里开始）

1. **里程碑 M-A（摄取质量）**：2.1 版面感知解析 → 2.2 表格结构化 →（清库重摄）跑前后检索对比。
   - 这是性价比最高、最快见效的一步，且改善纯文本检索。
2. **里程碑 M-B（多模态提质）**：2.3 图 caption 定位 + 图注关联、2.4 结构感知 chunking。
3. **里程碑 M-C（Agentic RAG）**：按 3.1→3.5 分步，先做 route + 查询分解，再多跳 + 反思。
4. （可选预研）M-D：公式提取、ColPali。

> 每个里程碑完成后：清库重摄（如涉及摄取改动）→ 跑 `scripts/evaluate.py` 看指标 → 保证 `pytest` 不回归。

## 7. 通用验收与约束

- 所有改动**保持现有可插拔架构**（工厂 + 惰性注册 + 配置开关），新方案作为新 provider 可切换、可回退。
- 摄取类改动需 `rm -rf data/db/{chroma,bm25,image_index.db,ingestion_history.db}` 清库重摄。
- 不破坏现有 **328 passed**；新功能补单测。
- 安全：沿用现有约束（路径白名单 / 防 SSRF / 限大小）；`config/settings.yaml` 的 DashScope Key 建议迁移到环境变量 `DASHSCOPE_API_KEY`。
