Agentic RAG 深度解析

一个交互式教程，带你理解「会自己思考的检索」——让 LLM 决定是否检索、检索什么、检索几轮、答得够不够，而不是一次盲目召回。

route → (transform → multi-hop retrieve → reflect)* → synthesize

🎯 总览：为什么需要 Agentic RAG WHY

经典 RAG 的流程是固定的：查询 → 一次向量检索 → 把片段塞给 LLM 生成。它有三个天然短板：①简单寒暄也照样检索，浪费算力；②复合问题（"对比 A 和 B"）一次召回抓不全；③召回不足时没有补救机制，直接拿着残缺上下文硬答。

Agentic RAG 把检索变成一个由 LLM 驱动的自主决策循环：先判断要不要检索、去哪个库检索；把复杂问题拆成子查询；多跳累积上下文；并在每跳后反思证据是否充分，不足则自己生成追问再补检；最后服务端合成带[n]引用的答案。

❌ 经典 RAG（query_knowledge_hub）

单向、无脑、一次召回，答案由外部客户端 LLM 拼

query → hybrid_search(top_k) → return passages # 不判断是否该检索 # 复合问题抓不全 # 召回不足无补救

✅ Agentic RAG（agentic_query）

闭环、自主、多跳，服务端合成带引用答案

route() # 要不要检索？去哪个库？ transform() # 拆成聚焦的子查询 retrieve()* # 多跳累积、去重 reflect()* # 不够？生成追问再补检 synthesize() # 合成 + [n] 引用
💡 关键洞察：Agentic RAG 的每一个"智能"环节（路由 / 改写 / 反思）都是最佳努力的优化项（best-effort）。任何一步的 LLM 调用或 JSON 解析失败，都会保守降级而非报错——这是它能在生产环境稳定运行的根本设计哲学。后面每一章都会反复看到这一点。

⚠️ 全局护栏：上面整段逻辑都被 run() 中的一个 try/except 包住。任何异常都会跌落到 _fallback()——退化成一次普通混合检索 + 尽力合成，并把 AgentResult.fallback=True。用户永远不会看到 agent 报错。

🧭 路由决策（Route） 3.1

QueryRouter 回答两个问题：①这个问题需不需要查知识库？②如果需要，查哪个 collection？

交互演示：不同问题的路由结果

白名单校验：防止幻觉 collection

LLM 提议的 collection 必须在可用集合内，否则被丢弃。这是一道安全闸门——防止 LLM 把查询路由到不存在或越权的库。

Python - QueryRouter._decide() # 白名单校验：丢弃任何不在 available 集合中的 collection proposed = data.get("collections", []) targets: list[str] = [] if isinstance(proposed, list): available_set = set(available_collections) for name in proposed: if isinstance(name, str) and name in available_set and name not in targets: targets.append(name)
💡 保守降级三连：① LLM 调用异常 → need_retrieval=True, collections=[]（检索全部）；② JSON 解析失败 → 同上（parse_fallback）；③ route_enabled=false → 直接默认检索全部。宁可多检索，绝不漏检索。

路由 Prompt 的核心规则

① 只有寒暄/闲聊/纯算术/无需文档支撑的常识才设 need_retrieval=false，存疑一律 true；
② collections 严格从 AVAILABLE 选，空数组 [] = 搜全部，绝不臆造；
③ 只输出裸 JSON，不要 ```json``` 代码块包裹。

✂️ 查询改写与分解（Transform） 3.2

口语化或复合的问题往往检索效果差。QueryTransformer 把它改写成一个或多个聚焦、关键词丰富的子查询——只有存在明确不同的需求时才分解。

❌ 原始复合查询

"BERT 和 GPT-4 在准确率和 推理速度上分别怎么样， 哪个更适合做线上服务？"

一次检索很难同时命中这么多子需求

✅ 分解后的子查询（≤ max_subqueries）

[ {"text": "BERT GPT-4 准确率对比"}, {"text": "BERT GPT-4 推理延迟 速度"}, {"text": "线上服务 模型选型 权衡"} ]

每条聚焦单一需求，各自检索

去重 + 截断（永不为空）

Python - QueryTransformer._transform() seen: set[str] = set() subqueries: list[SubQuery] = [] for item in data: text = self._extract_text(item) # 兼容裸字符串 / {"text": ...} if not text: continue key = text.strip().lower() if key in seen: continue # 归一化去重 seen.add(key) subqueries.append(SubQuery(text=text, purpose=...)) if len(subqueries) >= max_subqueries: break # 数量封顶
✅ 降级保证：LLM 失败、解析失败、或结果为空 → 一律回退为 [SubQuery(text=query)]（原始查询作为唯一子查询）。transform() 的返回值永远非空，所以检索总能进行。当 rewrite_enabled=false 时也直接走这条路。

🔍 多跳检索循环（Multi-hop Retrieve） 3.3

这是 Agentic RAG 的心脏：_retrieve_loop() 在子查询上迭代检索，去重累积上下文，受 max_hops 和 max_context_chunks 双重预算约束。反思（下一章）会把追问塞回 pending 触发新的一跳。

去重累积：按 chunk_id 留高分

不同子查询/不同跳可能召回同一片段。_accumulate() 用 chunk_id 去重，已存在则保留更高分的那次，且严格遵守 max_context_chunks 预算。

Python - AgenticRAG._accumulate() for r in results: existing = seen.get(r.chunk_id) if existing is not None: if r.score > context[existing].score: # 同片段保留高分 context[existing] = r continue if len(context) >= max_chunks: # 预算用尽，丢弃 continue seen[r.chunk_id] = len(context) context.append(r) added += 1 return added # 新增片段数

循环骨架

Python - AgenticRAG._retrieve_loop() max_hops = cfg.max_hops if cfg.multihop_enabled else 1 pending = [sq.text for sq in subqueries] hop = 0 while pending and hop < max_hops: hop += 1 current, pending = pending, [] for sub in current: results = self._retrieve(sub, ...) # 混合检索 results = self._maybe_rerank(sub, results) # 可选重排 new_hits += self._accumulate(context, seen, results, max_chunks) # …反思：不足则把追问写回 pending，触发下一跳…
💡 每个子查询检索 retrieval_top_k 条；可选地经 QueryReranker 重排（rerank.enabled 且重排器可用时）；重排失败绝不阻塞，直接返回原结果。

🤔 反思与自纠（Reflect） 3.4

每跳之后，Reflector（CRAG / Self-RAG 风格）评估：当前上下文够不够回答原始问题？不够就生成 1-3 个聚焦的追问，喂回多跳循环；够了就停。

反思触发的四个守卫条件

1

reflect_enabled 为真

总开关打开

2

context 非空

有东西才值得评估

3

hop < max_hops

最后一跳不反思——没有余地再行动了

4

reflect_rounds < max_reflect_rounds

反思轮数也有上限，防止反复 spin

Python - AgenticRAG._retrieve_loop()（反思片段） if (cfg.reflect_enabled and context and hop < max_hops and reflect_rounds < cfg.max_reflect_rounds): verdict = self._get_reflector().assess(query, context, trace=trace) if not verdict.sufficient and verdict.follow_up_queries: reflect_rounds += 1 pending = [q for q in verdict.follow_up_queries if q.strip().lower() not in asked] # 不重复已问过的

交互演示：充分 vs 不充分

⚠️ 保守即安全：反思的任何失败（LLM 异常 / JSON 解析失败 / 没给出具体追问）一律视为 sufficient=True——让循环停下来，而不是空转。这是成本与延迟的安全阀。Prompt 也明确要求"偏向 sufficient，过度追问伤延迟"。

🛡️ 容错与降级 RESILIENCE

分层降级：每个组件都"永不抛错"

🧭 路由降级

失败 → need_retrieval=True, collections=[]（检索全部）

✂️ 改写降级

失败 → 用原始查询作为唯一子查询，检索照常

🤔 反思降级

失败 → sufficient=True，循环停止，绝不空转

🔁 重排降级

重排器不可用/失败 → 返回原检索结果

🛟 全局降级

任何环节异常 → 一次混合检索 + 尽力合成，fallback=True

🔌 总开关降级

agent.enabled=false → 委托 query_knowledge_hub

全局降级链路

┌──────────────────────────────────────────────────────────┐
│  AgenticQueryTool.run(query)                              │
│                                                          │
│  ① agent.enabled = false ? ──YES──→ query_knowledge_hub    │
│       │ NO                              （行为不变）       │
│       ▼                                                  │
│  ② AgenticRAG.run() ── try ──→ route→…→synthesize        │
│       │                                                  │
│       └─ except 任意异常 ──→ _fallback()                   │
│                              · 一次 hybrid_search          │
│                              · 尽力 synthesize             │
│                              · fallback = True              │
└──────────────────────────────────────────────────────────┘
✅ 设计哲学：所有"智能"环节都是锦上添花的优化项，而非必经的关键路径。即使每一个 LLM 都罢工，系统也会优雅退化成一个最朴素但可用的 RAG。这正是它敢上生产的底气。

💡 一句话总结：Agentic RAG = 会思考的检索循环。它把"是否检索、检索什么、检索几轮、答得够不够"这些原本写死的决策交给 LLM 动态判断；同时用层层降级把每个 LLM 决策变成"能用则用、不能用就退"的最佳努力优化项。智能与稳健，二者兼得。
