这次解决了什么

原项目设计是 Image-to-Text（图→文字描述→文本检索），但端到端从未打通，图片数恒为 0。本次升级到 路径 B：把文本与图片编码进同一个向量空间，实现真正的「以文搜图 / 以图搜文 / 以图搜图」，并让 MCP 查询工具能直接接收图片提问。

维度升级前升级后
向量空间纯文本 text-embedding-v3图文统一 multimodal-embedding-v1（1024 维）
PDF 抽图占位（markitdown 只出文本，正则恒空）pypdfium2 真抽图落盘入库
图片描述硬编码 Azure，未启用qwen-vl-max 经工厂调用，默认开启
image_refs从未被填充（隐藏断点）抽图→分配→缝描述→还原原图 全贯通
图片检索无图片作为独立向量记录入同一 collection
query 接收图片仅 query:string新增 image 入参（base64/受限路径）
已真实验证：文本向量与图片向量均落 1024 维同一空间；GPT-3 论文真抽出 83 张图、Transformer 3 张图，占位与抽图数一致。

三种查询模式

纯文本 → Dense(文本) + Sparse(BM25) + RRF（原行为）
纯图片 → Dense(图片向量) only（BM25 对图无意义，跳过）
图文混合 → Dense 用图向量 + Sparse 用文本关键词 + RRF

安全设计

风险防护
路径穿越本地图片路径必须 resolve() 后位于 data/ 白名单目录内，拒绝 .. 越权
SSRF查询图片拒绝远程 URL（http/https/ftp）
超大载荷Base64/文件解码后限制 ≤ 10 MiB；返回图片单张 ≤ 5 MiB
可靠性单图抽取/编码/描述失败均 warning 跳过，绝不阻断文本答案
路径来源还图时路径仅来自 ImageStorage 索引，不接受调用方任意路径
